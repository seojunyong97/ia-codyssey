# 3. 실행 로그(대화 로그)

### 1) 시나리오 설명

- **테스트 안건:** 영화 단편 <새벽의 문> 시나리오 회의록 요약 및 프리 프로덕션 일정 관리
- **시나리오 목표:** 초기 회의록 요약본 도출 후, 실무 현장에서 실시간으로 발생하는 변동 상황에 시스템 프롬프트가 얼마나 견고하게 대처하는지 검증합니다.
- **주요 검증 상황:**
    1. **추가 정보의 사후 제공:** 정PD의 카톡 피드백을 통해 미결 상태였던 소품비 초과 예산 수치(30만 원)가 뒤늦게 확정되는 상황 부여 (가이드라인 요구사항 충족)
    2. **중간 조건 및 정책 변경:** 지도교수님의 피드백으로 인해 이미 합의되었던 엔딩 수정 결정사항이 번복되고 다시 '미결 이슈'로 돌아가는 상황 부여 (가이드라인 요구사항 충족)
    3. **모호한 입력 처리:** 사용자가 주어, 기한, 장소 없이 "식당 예약 좀 해달라"고 모호하게 지시했을 때, 시스템 프롬프트 규칙 규칙 #5에 따라 임의로 환각(Hallucination)을 일으키지 않고 사용자에게 역으로 부족한 정보를 되묻는지 검증

### 2) 문제 지점과 개선 요약

- **문맥 유지 성능 (Pass):** 10턴이 넘어가는 긴 대화 속에서도 시스템 프롬프트에 지정된 10년 차 영화 제작 PM 페르소나와 '4개 섹션(이모지 포함)' 출력 형식을 한 번도 깨뜨리지 않고 일관되게 유지했습니다.
- **조건 변경 유연성 (Pass):** 추가 예산 정보가 입력되었을 때 '미결 이슈'에서 해당 항목을 삭제하고 '핵심 결정사항'으로 지능적으로 이동시켰으며, 엔딩 번복 상황 시 변경된 부분만 정확히 인지하여 문서를 원복하고 나머지 정보는 그대로 유지하는 높은 컨텍스트 유지력을 보였습니다.
- **환각 방지 및 예외 처리 (Pass):** 입력 데이터가 불명확한 식당 예약 지시(Turn 6) 상황에서 가상의 식당 이름을 지어내거나 일정을 임의로 확정 짓지 않고, 필요한 메타데이터(날짜, 인원, 위치 등)를 명확히 되물어 엣지 케이스를 안전하게 방어했습니다.
